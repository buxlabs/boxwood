# Prose

`Prose` renders markdown content with a safe templating layer: interpolation,
conditionals, loops and custom components. It is designed for content that
lives close to data - blog posts, product pages, CMS-driven sections.

```js
const { compile } = require("boxwood")
const { Prose } = require("boxwood/ui")

const template = (data) =>
  Prose({ data }, `
# {title}

Published {date.toLocaleDateString('en-US')}

{#each posts.slice(0, 3) as post, i}
Part {i + 1}: **{post.title ?? 'Untitled'}**
{/each}
`)
```

`Prose(params, children)`:

- `params.data` - object the expressions resolve against
- `params.components` - custom components usable as tags in the content
- any other key (e.g. `class`) - applied as an HTML attribute to rendered elements
- `children` - the markdown string (arrays of strings are rendered one by one)

Unresolved expressions keep their `{placeholder}` text - nothing throws at
render time. Use the [validator](#validation) to surface problems.

## Markdown

| Feature | Syntax |
| --- | --- |
| Headings | `# h1` … `###### h6` |
| Lists | `- item`, `1. item`, nested by indentation |
| Task lists | `- [ ] todo`, `- [x] done` |
| Blockquotes | `> quote` |
| Horizontal rule | `---`, `***`, `___` |
| Code blocks | ` ```js ` … ` ``` ` |
| Inline | `**bold**`, `*italic*`, `~~strikethrough~~`, `` `code` ``, `[text](url)`, `![alt](src)`, `<https://autolink>` |
| Tables | see [Tables](#tables) |

### Task lists

```
- [ ] write the draft
- [x] pick a title
```

Items render with a disabled checkbox (`<input type="checkbox" disabled checked>`),
work in ordered lists too (`1. [x] shipped`) and accept `x` or `X`. The item
text supports expressions and inline markdown.

### Strikethrough

`~~text~~` renders as `<del>text</del>` and nests with other inline syntax
(`~~**bold**~~`). Single tildes are plain text (`~5 minutes`), and `\~` escapes
a tilde.

### Heading anchors

Headings automatically get an `id` derived from their visible text:

```
## Mój tytuł          ->  <h2 id="moj-tytul">Mój tytuł</h2>
## Using `slice`      ->  <h2 id="using-slice">…</h2>
```

- diacritics are transliterated (`ż -> z`, `ł -> l`, `ß -> ss`)
- inline markdown syntax is stripped before slugging (`[Link](url)` slugs as `link`)
- repeated headings get `-2`, `-3` suffixes
- an explicit `id` param on `Prose` wins - no anchor is generated then
- anchors are a **Prose** feature; the plain `Markdown` component leaves
  headings untouched (`<h2>…</h2>`), staying a faithful GFM renderer. Tables,
  task lists and strikethrough are standard GFM and appear in both.

### Tables

```
| Name | Price |
| :--- | ---: |
| Widget | 10 |
```

- a header row plus a `| --- |` separator row starts a table; a `|` line
  without a separator below it stays a paragraph
- `:---`, `:---:`, `---:` render as `text-align` styles (left is the default
  and emits no style)
- cells support inline markdown and expressions
- short rows are padded to the header width
- escape a literal pipe with `\|`; pipes inside `` `inline code` `` need no escape
- rows can be generated by loops:

```
| Name | Price |
| --- | --- |
{#each products as product}| {product.name} | {product.price} |
{/each}
```

### Code is literal

Fenced code blocks and inline code are never touched by templating -
`{name}` inside code renders as-is. Code blocks with a language render as
`<pre><code class="language-js">`, ready for Prism or highlight.js.

## Expressions

Expressions appear inside `{…}` in text, attributes and control blocks.

### Paths

```
{name}  {user.name}  {images[0].src}  {items.length}
```

### Methods

Only whitelisted non-mutating methods can be called:

`slice` `at` `concat` `includes` `indexOf` `lastIndexOf` `join`
`toUpperCase` `toLowerCase` `trim` `charAt` `substring` `split` `padStart`
`padEnd` `repeat` `replace` `replaceAll` `startsWith` `endsWith` `toFixed`
`toLocaleDateString` `toLocaleTimeString` `toLocaleString` `toISOString`
`toString`

Arguments are literals (numbers, quoted strings, booleans, `null`) or
expressions resolved against data:

```
{images.slice(0, 2)}
{images.slice(0, limit)}
{text.slice(0, text.indexOf(' '))}
{post.date.toLocaleDateString('pl-PL')}
```

Anything else - unknown methods, unparseable arguments - resolves to
`undefined` and keeps the placeholder. Method calls never mutate data.

### Array literals and spread

```
{[images[0], images[2]]}
{[...featured, ...latest]}
{[...images.slice(0, 1), "static.jpg"]}
```

Spreading a non-array keeps the value as a single element; spreading
`null`/`undefined` adds nothing.

### Fallbacks: `??` and `||`

```
{name ?? "Guest"}          fallback for null/undefined only
{title || "Untitled"}      fallback for any falsy value ("" and 0 included)
{nickname ?? name ?? "Guest"}
```

### Logical `&&`

```
{user.active && user.name || "Anonymous"}
```

JS semantics and precedence: `&&` binds tighter than `||`. Mixing `??` with
`||` or `&&` in one expression is invalid (as in JS, where it requires
parentheses - which Prose does not support); such expressions resolve to
`undefined` and the validator reports them.

### Arithmetic

```
{i + 1}  {items.length - 1}  {price * quantity}  {total / count}
```

- operators must be surrounded by spaces: `{i + 1}` works, `{i+1}` does not
- `*` and `/` bind tighter than `+` and `-`; chains evaluate left to right
- `+` also concatenates strings: `{first + ' ' + last}`
- a nullish operand or NaN result resolves to `undefined` (placeholder stays -
  "NaN" never renders)

### Escaping

`\{` renders a literal brace. `{` without a matching `}` is left as text.

## Conditionals

```
{#if user.premium}
Exclusive content
{#elseif user.registered}
Regular content
{#else}
Sign up!
{/if}
```

Conditions support:

- truthiness: `{#if title}` - `null`, `undefined`, `""`, `0`, `false` and
  **empty arrays** are falsy (note: `[]` is truthy in JS but falsy here)
- negation: `{#if !hidden}`
- comparisons: `>` `<` `>=` `<=` `==` `!=` `===` `!==`
- logical operators with JS precedence:
  `{#if articles.length === 0 || user.role === 'admin'}`
- full expressions on both sides:
  `{#if items.length - 1 > 0}`, `{#if title ?? subtitle}`,
  `{#if posts.slice(0, limit).length == limit}`, `{#if tags.includes(tag)}`

## Loops

```
{#each posts as post, i}
Part {i + 1} of {posts.length}: {post.title}
{#else}
No posts yet.
{/each}
```

- `{#each items}` - current item is `{item}`
- `{#each items as post}` - custom name
- `{#each items as post, i}` - zero-based index
- `{#else}` renders when the array is empty or missing
- the array part is any expression:
  `{#each posts.slice(0, limit) as post}`,
  `{#each [...featured, ...latest] as post}`,
  `{#each posts ?? [] as post}`
- loops nest; inner loops can reference the outer item

## Comments

```
{!-- not rendered, may span
multiple lines --}
```

## Custom components

```js
const Gallery = ({ images = [] }) =>
  Div({ class: "gallery" }, images.map((image) => Img({ src: image })))

Prose({ data, components: { Gallery } }, content)
```

```
<Gallery images="{images.slice(0, limit)}" />

<Gallery images="{[
  ...featured,
  images[0]
]}" />

<Card title="Photo of {name}">
Markdown content becomes the component's children.
</Card>
```

- attribute values `"{expression}"` resolve to the raw value (array, object,
  number), not a string
- other attributes interpolate partially: `href="/posts/{slug}"`
- quoted attribute values may contain `>`: `title="5 > 3"`
- tags can span multiple lines; a blank line ends the candidate tag, so plain
  text that happens to start with a component name stays text - which also
  means attributes themselves cannot contain blank lines
- safe builtin HTML tags (`div`, `img`, `table`, …) and the UI components
  `Center`, `Container`, `Grid`, `Group`, `Stack` are always available

## Validation

Rendering never throws on template problems - run the validator in tests or
in the authoring pipeline instead:

```js
const { Prose } = require("boxwood/ui")

const issues = Prose.validate(content, {
  data: { title: "…" },        // optional - enables unknown-variable checks
  components: { Gallery },     // optional - enables unknown-component checks
})
// [{ line: 3, type: "unknown-variable", message: "Unknown variable: titel" }]
```

Issue types: `unknown-variable`, `unknown-component`, `unsafe-method`,
`forbidden-property`, `malformed-expression`, `malformed-block`,
`unmatched-block`, `unclosed-block`, `unclosed-code-block`,
`unclosed-comment`.

Forgotten values are reported too: empty braces (`{}`, also inside attribute
values) and empty conditions (`{#if }`, `{#elseif }`). The renderer keeps
them as literal text - escape braces (`\{`) or use inline code for
intentional literal `{}`.

## Safety

Prose is built to render untrusted content without executing it:

- **No code execution.** Expressions resolve against `data` only; method
  calls are limited to a non-mutating whitelist. There is no `eval`.
- **No prototype access.** `constructor`, `__proto__` and `prototype` never
  resolve, so `{x.constructor.constructor}` cannot reach `Function` or
  pollute prototypes (reported as `forbidden-property`).
- **Script URLs neutralized.** Markdown links with a `javascript:`, `data:`
  or `vbscript:` scheme render as `href="#"`.
- **Bounded work.** Expressions over 1000 characters and nesting deeper than
  50 levels resolve to `undefined` instead of exhausting memory or the stack,
  so a malicious `{[[[…]]]}` cannot hang the renderer.

Note that variable *values* are still trusted: HTML escaping applies to
interpolated text, but a value you place into a `raw()` node or an
`href`/`src` you build yourself is your responsibility.

## Limitations

- no parentheses for grouping in expressions - layer with `||`/`&&`/`??`
  hierarchies instead
- no object literals in expressions
- no ternary operator - use `{#if}` blocks or `&&`/`||` chains
- tables inside blockquotes are not parsed
- expressions cannot contain `{` or `}`
